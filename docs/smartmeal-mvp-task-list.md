# SmartMeal MVP 开发任务清单

## 1. 任务拆分原则

本任务清单面向 MVP 首版开发，优先完成可演示闭环：

AI 对话输入 -> 今日三餐推荐 -> 营养反馈 -> 库存联动 -> 购物清单。

每个任务都应尽量小，完成后可单独验收。

## 2. 前端任务

### FE-01 初始化前端项目

说明：

- 使用 React + TypeScript + Vite。
- 配置 UTF-8 源码文件。
- 建立基础目录结构。

建议目录：

```text
src/
  app/
  components/
  data/
  features/
  types/
  utils/
```

验收：

- 项目可以启动。
- 首页显示 SmartMeal 基础页面。

### FE-02 建立类型定义

说明：

- 定义用户、库存、餐单、营养、购物清单、聊天消息类型。
- 类型与 `docs/smartmeal-mvp-dev-breakdown.md` 保持一致。

验收：

- 所有 mock 数据通过 TypeScript 类型检查。

### FE-03 建立 mock 数据

说明：

- 准备示例用户营养目标。
- 准备示例库存。
- 准备示例三餐。
- 准备示例购物清单。
- 准备示例 AI 对话。

验收：

- 页面首次加载不为空。
- mock 数据可以支撑全部核心模块。

### FE-04 实现首页框架

说明：

- 实现顶部标题和导航。
- 实现三栏桌面布局。
- 左栏 AI 对话，中栏今日三餐，右栏库存和购物清单。

验收：

- 1440px 宽度下三栏展示清晰。
- 主要模块不重叠、不溢出。

### FE-05 实现 AI 对话区

说明：

- 展示用户和 AI 消息。
- 支持输入文本并发送。
- 支持快捷操作按钮。
- 发送后展示 mock AI 回复。

验收：

- 用户输入会追加到消息列表。
- 快捷操作会触发对应 mock 状态变化。

### FE-06 实现今日三餐卡片

说明：

- 展示早餐、午餐、晚餐。
- 每张卡展示图片、名称、描述、时间、营养数据。
- 支持展开详情。
- 支持“换一个”按钮。

验收：

- 展开只影响当前餐卡。
- “换一个”只替换当前餐。

### FE-07 实现营养概览

说明：

- 展示目标值、实际值、百分比进度。
- 展示热量、蛋白质、碳水、脂肪、膳食纤维。
- 展示 AI 微调建议。

验收：

- 三餐数据变化后，营养概览同步变化。

### FE-08 实现库存模块

说明：

- 展示库存总数、临期数、需采购数。
- 展示库存表格。
- 支持新增库存食材。

验收：

- 新增食材后库存列表更新。
- 临期状态正确展示。

### FE-09 实现购物清单模块

说明：

- 按分类展示购物清单。
- 支持勾选已购买。
- 显示每个购物项的购买原因。

验收：

- 勾选状态即时更新。
- 分类展示清晰。

## 3. 后端任务

### BE-01 初始化后端项目

说明：

- 使用 Node.js + Express 或 Fastify。
- 配置 UTF-8。
- 建立基础 API 路由。

验收：

- `/api/health` 返回正常。

### BE-02 实现用户配置 API

接口：

- `GET /api/profile`
- `PATCH /api/profile`

验收：

- 可以读取和更新营养目标、口味偏好、饮食限制。

### BE-03 实现库存 API

接口：

- `GET /api/inventory`
- `POST /api/inventory`
- `PATCH /api/inventory/:id`
- `DELETE /api/inventory/:id`

验收：

- 支持库存增删改查。
- 可根据过期日期计算临期状态。

### BE-04 实现购物清单生成 API

接口：

- `POST /api/shopping-list/generate`

验收：

- 能根据餐单食材和库存生成缺口清单。
- 库存充足的食材不重复加入购物清单。

### BE-05 实现 AI 餐单生成 API

接口：

- `POST /api/chat/meal-plan`

验收：

- 输入自然语言需求后返回结构化餐单。
- 返回内容包含 reply、meals、nutritionSummary、shoppingList、inventoryUsage、suggestions。
- AI 失败时返回可用的 fallback 餐单。

## 4. AI 与推荐逻辑任务

### AI-01 设计结构化输出 Schema

说明：

- 约束 AI 必须输出固定 JSON。
- 字段必须覆盖前端渲染所需数据。

验收：

- 缺少关键字段时能被 schema 校验捕获。

### AI-02 设计 MVP 提示词

说明：

- 提示词需要包含用户目标、库存、饮食偏好、输出格式。
- 明确要求优先使用临期和库存食材。

验收：

- 多次生成结果结构稳定。
- AI 不返回无法解析的大段自由文本作为主数据。

### AI-03 实现营养估算

说明：

- MVP 可先使用菜品级估算数据。
- 后续再升级到食材级营养数据库。

验收：

- 每餐和每日都有热量、蛋白质、碳水、脂肪、膳食纤维。

### AI-04 实现推荐后处理

说明：

- 对 AI 结果进行字段补全、异常过滤、默认值处理。
- 生成最终购物清单和营养差异。

验收：

- 前端永远拿到可渲染结构。

## 5. 测试任务

### QA-01 前端布局测试

范围：

- 1440px 桌面。
- 1280px 桌面。
- 1024px 小桌面。

验收：

- 无明显重叠。
- 按钮和输入框可点击。
- 文字不溢出核心容器。

### QA-02 核心流程测试

用例：

- 输入“今天清淡，多用鸡蛋和番茄”。
- 点击“提升蛋白质”。
- 点击“多用库存食材”。
- 新增库存食材。
- 勾选购物清单。

验收：

- 所有操作都有明确 UI 反馈。
- 三餐、营养、购物清单状态保持一致。

### QA-03 API 合约测试

范围：

- profile API。
- inventory API。
- meal-plan API。
- shopping-list API。

验收：

- 响应字段和前端类型一致。
- 错误响应有明确 message。

## 6. 推荐开发顺序

1. FE-01 初始化前端项目。
2. FE-02 建立类型定义。
3. FE-03 建立 mock 数据。
4. FE-04 实现首页框架。
5. FE-05 到 FE-09 完成前端交互闭环。
6. BE-01 到 BE-04 完成基础后端。
7. AI-01 到 AI-04 接入 AI 推荐逻辑。
8. BE-05 替换 mock 餐单生成。
9. QA-01 到 QA-03 做验收。

## 7. 首次可演示版本标准

达到以下条件即可作为第一次 Demo：

- 用户能在 AI 对话框输入需求。
- 页面能生成或切换今日三餐。
- 用户能看到今日营养目标完成情况。
- 用户能看到库存被使用的提示。
- 用户能看到购物清单，并勾选已购买。
- 页面整体视觉接近 PRD 示意图。
